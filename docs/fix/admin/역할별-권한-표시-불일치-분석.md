# 역할별 권한 표시 불일치 분석

## 사용자 질문

관리자 역할·권한 관리 화면에서 특정 회원에게 `ADMIN_USER` 역할을 저장했다.
다른 역할의 권한은 건드리지 않았는데, 아래 `역할별 권한 편집` 영역에서 `BUSINESS`를 선택하면 `READ`, `WRITE`, `DELETE`, `MANAGE`가 모두 부여된 것처럼 보인다.

확인해야 할 점:

- API가 `BUSINESS` 권한을 잘못 내려주는 것인지
- 프론트가 `BUSINESS` 권한을 잘못 표기하는 것인지

## 화면에서 관찰된 현상

- 상단 `멤버 역할 부여` 영역:
  - 검색된 회원은 `vhkdvhkhdrb`
  - 회원 표시 타입은 `MEMBER`
  - 선택 역할은 `ADMIN_USER`
  - 현재 보유 권한은 `READ`, `WRITE`, `DELETE`, `MANAGE`
- 하단 `역할별 권한 편집` 영역:
  - `BUSINESS` 탭을 선택했을 때 `BUSINESS — 부여된 권한 (4)`로 표시됨
  - 권한 칩으로 `READ`, `WRITE`, `DELETE`, `MANAGE`가 표시됨

## 프론트 코드 근거

### 하단 역할별 권한 편집

하단 `RolePermissionPanel`은 선택된 역할 ID로 역할 상세 API를 호출한다.

- `app/(manager)/manager/role/page.tsx`
  - `selectedRoleId`가 바뀌면 `loadDetail(selectedRoleId)` 호출
  - `loadDetail`은 `AuthorizationService.getRole(roleId)` 결과를 `roleDetail`에 저장
  - 화면은 `roleDetail.permissions`를 그대로 렌더링

관련 코드 흐름:

```ts
setRoleDetail(await AuthorizationService.getRole(roleId));
```

```tsx
{roleDetail.permissions.map((p) => (
  <li key={String(p.name)}>
    {String(p.name)}
  </li>
))}
```

API 래퍼:

- `lib/api/admin/authorization.ts`

```ts
getRole: (roleId: string) =>
  unwrap<RoleDetail>(api.get(`/api/v1/admin/authorization/roles/${roleId}`)),
```

따라서 하단 `BUSINESS` 권한 4개 표시는 프론트가 하드코딩하거나 `memberType`에서 파생한 값이 아니라,
`GET /api/v1/admin/authorization/roles/{roleId}` 응답의 `permissions` 배열을 표시한 것이다.

### 상단 역할 선택 카드

상단 역할 목록의 권한 뱃지도 최초 로딩 시 각 역할 상세 API를 호출해 만든 `rolePermMap`에서 나온다.

```ts
AuthorizationService.getRole(r.id!).then((d) => ({
  name: r.name,
  perms: d.permissions.map((p) => String(p.name)),
}))
```

다만 이 `rolePermMap`은 최초 로딩 후 하단에서 권한을 추가/삭제해도 자동 갱신되지 않는다.
그래서 상단 카드의 뱃지는 stale 상태가 될 수 있다. 하지만 이번 질문의 핵심인 하단 `BUSINESS` 표시 자체는
`roleDetail.permissions`를 다시 조회한 결과다.

## 백엔드/OpenAPI 근거

OpenAPI 기준:

- `GET /api/v1/admin/authorization/roles/{roleId}`
  - 역할 상세 조회
  - 응답은 `RoleDetailResponse`
  - `permissions`는 “역할에 할당된 권한 목록”

백엔드 서비스 흐름:

- `AdminAuthorizationService.findRoleDetail(roleId)`
  - `roleRepository.findById(roleId)`로 역할 조회
  - `rolePermissionRepository.findPermissionsByRoleId(roleId)`로 역할에 연결된 권한 조회
  - `RoleDetailResponse.of(role, permissions)`로 응답 생성

즉 백엔드도 `auth.role_permission` 매핑을 조회해 그대로 내려주는 구조다.

## DB 초기값 근거

백엔드 Liquibase 초기 마이그레이션 의도는 다음과 같다.

- `MEMBER`: `READ`
- `BUSINESS`: `READ`, `WRITE`
- `ADMIN_USER`: `READ`, `WRITE`
- `ADMIN_MASTER`: `READ`, `WRITE`, `DELETE`, `MANAGE`
- `ADMIN_DEVELOPER`: `READ`, `WRITE`, `DELETE`, `MANAGE`

파일:

- `/Users/hig/IdeaProjects/reservation/modules/iam/src/main/resources/db/changelog/V002__create_permission_tables.xml`

따라서 정상 초기 상태라면 `BUSINESS`는 2개 권한만 가져야 한다.

## 현재까지의 판단

가장 가능성이 높은 원인:

1. 실행 중인 API가 `BUSINESS` 역할 상세 응답에 `READ`, `WRITE`, `DELETE`, `MANAGE`를 내려주고 있다.
2. 그 API가 읽는 실행 DB의 `auth.role_permission`에 `BUSINESS`와 `DELETE`, `MANAGE` 매핑이 실제로 들어가 있다.

프론트 표시 오류 가능성은 낮다.
하단 패널에는 `BUSINESS` 권한을 임의로 4개로 만드는 프론트 로직이 보이지 않는다.

## 아직 확인하지 못한 것

실행 중인 서버/DB의 실제 응답과 테이블 내용은 아직 직접 확인하지 않았다.
아래 둘 중 하나를 확인하면 원인을 확정할 수 있다.

### 1. API 응답 확인

브라우저 개발자 도구 Network에서 `BUSINESS`를 선택할 때 호출되는 요청을 확인한다.

```http
GET /api/v1/admin/authorization/roles/{businessRoleId}
```

응답 예시에서 `data.permissions`가 4개면 API/DB 쪽 상태 문제다.

### 2. DB 매핑 확인

실행 DB에서 아래 쿼리를 확인한다.

```sql
select role_name, permission_name
from auth.role_permission
where role_name = 'BUSINESS'
order by permission_name;
```

기대값:

```text
BUSINESS | READ
BUSINESS | WRITE
```

현재 화면과 같은 상태라면 다음처럼 나올 가능성이 높다.

```text
BUSINESS | DELETE
BUSINESS | MANAGE
BUSINESS | READ
BUSINESS | WRITE
```

## 관련해서 별도로 발견한 프론트 타입 이슈

`MemberRoleSummary` 프론트 타입은 `roles: RoleSummary[]`로 선언되어 있다.

```ts
export interface MemberRoleSummary {
  memberId: string;
  roles: RoleSummary[];
  permissions: PermissionSummary[];
}
```

하지만 OpenAPI/백엔드 DTO는 `role` 단수 필드다.

```yaml
MemberRoleResponse:
  properties:
    memberId: ...
    role: ...
    permissions: ...
```

이 타입 불일치는 상단 “선택된 멤버의 현재 역할 선택값” 표시/저장 흐름에 영향을 줄 수 있다.
다만 하단 `BUSINESS` 권한 4개 표시의 직접 원인은 아니다.

## 다음 작업 제안

1. 실행 API에서 `GET /api/v1/admin/authorization/roles/{businessRoleId}` 응답을 확인한다.
2. 실행 DB의 `auth.role_permission`에서 `BUSINESS` 매핑을 확인한다.
3. DB가 잘못되어 있으면 `BUSINESS`의 `DELETE`, `MANAGE` 매핑을 제거하는 마이그레이션 또는 운영 데이터 보정이 필요하다.
4. 별도 작업으로 프론트 `MemberRoleSummary.roles`를 백엔드 계약에 맞게 `role`로 수정한다.
