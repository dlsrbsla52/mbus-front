# 역할별 권한 표시 불일치 확인 내용

## 확인 대상

관리자 역할·권한 관리 화면에서 특정 회원에게 `ADMIN_USER` 역할을 저장했는데, 하단 `역할별 권한 편집` 영역에서 `BUSINESS`를 선택하면 `READ`, `WRITE`, `DELETE`, `MANAGE`가 모두 부여된 것처럼 보이는 현상을 확인했다.

핵심 질문은 다음 두 가지였다.

- API가 `BUSINESS` 권한을 잘못 내려주는지
- 프론트가 `BUSINESS` 권한을 잘못 표기하는지

## 결론

하단 `역할별 권한 편집` 영역은 **선택한 회원의 권한**을 보여주는 영역이 아니다.

이 영역은 **선택한 역할 자체에 DB로 매핑된 권한**을 조회해서 보여준다.

따라서 `BUSINESS` 탭을 선택했을 때 `READ`, `WRITE`, `DELETE`, `MANAGE`가 모두 보인 이유는, 현재 실행 DB의 `auth.role_permission`에 `BUSINESS` 역할과 네 권한이 실제로 매핑되어 있기 때문이다.

프론트가 계정별 권한을 `BUSINESS` 권한처럼 잘못 표시한 것은 아니다.

### 상단 멤버 역할 부여 영역과의 차이

상단 `멤버 역할 부여` 영역은 특정 회원의 역할과 권한을 조회한다.

```ts
const summary = await AuthorizationService.getMemberRole(member.id);
```

반면 하단 `역할별 권한 편집` 영역은 회원과 무관하게 역할 자체를 조회한다.

따라서 특정 회원 `vhkdvhkdrb`에게 `ADMIN_USER`를 저장한 것과, 하단에서 `BUSINESS`를 선택했을 때 표시되는 권한은 직접적인 관계가 없다.

## 백엔드 확인 내용

파일:

- `/Users/hig/IdeaProjects/reservation/modules/iam/src/main/kotlin/com/media/bus/iam/admin/controller/AdminAuthorizationController.kt`
- `/Users/hig/IdeaProjects/reservation/modules/iam/src/main/kotlin/com/media/bus/iam/admin/service/AdminAuthorizationService.kt`
- `/Users/hig/IdeaProjects/reservation/modules/iam/src/main/kotlin/com/media/bus/iam/auth/repository/RolePermissionRepository.kt`

역할 상세 API는 다음 컨트롤러에서 처리한다.

```kotlin
@GetMapping("/roles/{roleId}")
fun getRoleDetail(@PathVariable roleId: UUID): ApiResponse<RoleDetailResponse> =
    ApiResponse.success(adminAuthorizationService.findRoleDetail(roleId))
```

서비스는 역할을 조회한 뒤, 해당 역할 ID에 매핑된 권한을 조회한다.

```kotlin
fun findRoleDetail(roleId: UUID): RoleDetailResponse {
    val role = roleRepository.findById(roleId)
        ?: throw BaseException(AuthResult.ROLE_NOT_FOUND)

    val permissions = rolePermissionRepository.findPermissionsByRoleId(roleId)

    return RoleDetailResponse.of(role, permissions)
}
```

Repository는 `auth.role_permission`을 기준으로 권한을 조회한다.

```kotlin
fun findPermissionsByRoleId(roleId: UUID): List<PermissionEntity> =
    PermissionEntity.wrapRows(
        (PermissionTable innerJoin RolePermissionTable)
            .select(PermissionTable.columns)
            .where { RolePermissionTable.roleId eq EntityID(roleId, RoleTable) }
    ).toList()
```

따라서 API도 회원별 임시 권한을 만드는 것이 아니라, DB의 역할-권한 매핑을 그대로 내려준다.

## 초기 마이그레이션 기준

파일:

- `/Users/hig/IdeaProjects/reservation/modules/iam/src/main/resources/db/changelog/V002__create_permission_tables.xml`

초기 시드 의도는 다음과 같다.

- `MEMBER`: `READ`
- `BUSINESS`: `READ`, `WRITE`
- `ADMIN_USER`: `READ`, `WRITE`
- `ADMIN_MASTER`: `READ`, `WRITE`, `DELETE`, `MANAGE`
- `ADMIN_DEVELOPER`: `READ`, `WRITE`, `DELETE`, `MANAGE`

즉 정상 초기 상태만 놓고 보면 `BUSINESS`는 `READ`, `WRITE` 두 개 권한만 가져야 한다.

## 실행 DB 확인 결과

현재 실행 중인 PostgreSQL 컨테이너 `msa-postgres`의 DB를 직접 조회했다.

확인 쿼리:

```sql
select r.name as role_name, p.name as permission_name
from auth.role_permission rp
join auth.role r on r.id = rp.role_id
join auth.permission p on p.id = rp.permission_id
where r.name = 'BUSINESS'
order by p.name;
```

조회 결과:

```text
 role_name | permission_name
-----------+-----------------
 BUSINESS  | DELETE
 BUSINESS  | MANAGE
 BUSINESS  | READ
 BUSINESS  | WRITE
(4 rows)
```

따라서 현재 하단 화면의 `BUSINESS — 부여된 권한 (4)` 표시는 실행 DB 상태와 일치한다.

## 추가로 확인된 DB 상태

전체 역할-권한 매핑을 확인했을 때 `BUSINESS`뿐 아니라 `ADMIN_USER`, `MEMBER`도 초기 시드와 다른 권한을 갖고 있었다.

주요 결과:

```text
ADMIN_USER | DELETE
ADMIN_USER | MANAGE
ADMIN_USER | READ
ADMIN_USER | WRITE

BUSINESS   | DELETE
BUSINESS   | MANAGE
BUSINESS   | READ
BUSINESS   | WRITE

MEMBER     | DELETE
MEMBER     | MANAGE
MEMBER     | READ
MEMBER     | WRITE
```

`BUSINESS`의 `DELETE`, `MANAGE` 매핑은 초기 시드 UUID가 아니라 새 UUID였고, 생성 시각은 다음과 같았다.

```text
BUSINESS | DELETE | 2026-06-04 23:42:40+09
BUSINESS | MANAGE | 2026-06-04 23:42:41+09
```

초기 시드 데이터는 `2026-03-21`에 생성되어 있었다.

이 점을 보면 `BUSINESS`의 추가 권한은 초기 마이그레이션으로 들어간 값이 아니라, 이후 관리 API 호출 또는 수동 DB 조작으로 추가된 데이터일 가능성이 높다.

## 회원 역할 확인 쿼리와 의미

사용자가 제시한 쿼리:

```sql
SELECT *
FROM auth.member as m
JOIN auth.member_role as mr on m.id = mr.member_id
WHERE m.login_id = 'vhkdvhkdrb';
```

이 쿼리는 특정 회원이 어떤 역할을 갖는지 확인하는 데 유효하다.

하지만 이 쿼리만으로는 해당 역할이 어떤 권한을 갖는지까지는 확인할 수 없다. 역할의 실제 권한까지 보려면 `auth.role_permission`, `auth.role`, `auth.permission`을 추가로 조인해야 한다.

예시:

```sql
select
  m.login_id,
  r.name as role_name,
  p.name as permission_name
from auth.member m
join auth.member_role mr on mr.member_id = m.id
join auth.role r on r.id = mr.role_id
join auth.role_permission rp on rp.role_id = r.id
join auth.permission p on p.id = rp.permission_id
where m.login_id = 'vhkdvhkdrb'
order by r.name, p.name;
```

## 정리

- 하단 `역할별 권한 편집`은 계정별 권한 표시 영역이 아니다.
- 하단 영역은 선택한 역할의 DB 매핑 권한을 보여주는 영역이다.
- 현재 `BUSINESS`가 4개 권한으로 보이는 이유는 실행 DB에 `BUSINESS -> READ, WRITE, DELETE, MANAGE`가 실제로 들어 있기 때문이다.
- 프론트가 `BUSINESS` 권한을 임의로 4개로 만든 증거는 없다.
- 초기 마이그레이션 기준으로는 `BUSINESS`가 `READ`, `WRITE`만 가져야 하므로, 현재 DB 데이터는 초기 의도와 다르다.
- `BUSINESS`의 추가 권한은 생성 시각상 이후 관리 API 호출 또는 수동 DB 조작으로 추가된 데이터일 가능성이 높다.

## 후속 확인 또는 조치 후보

1. 현재 운영 의도상 `BUSINESS`가 `READ`, `WRITE`만 가져야 하는지 정책을 확정한다.
2. 정책상 2개 권한이 맞다면 `BUSINESS`의 `DELETE`, `MANAGE` 매핑을 제거한다.
3. 같은 기준으로 `MEMBER`, `ADMIN_USER`도 현재 DB 권한이 의도와 맞는지 점검한다.
4. 운영 데이터 보정은 수동 DML보다 Liquibase changelog로 남기는 것이 프로젝트 규칙에 맞다.
5. 프론트 문구가 혼동을 줄 수 있으므로 하단 영역 제목을 “역할 자체 권한 편집”처럼 명확히 바꾸는 것도 고려할 수 있다.
