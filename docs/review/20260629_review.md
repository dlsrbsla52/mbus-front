# mbus-front 소스 리뷰 (2026-06-29)

## 요약

- 리뷰 범위: Next.js App Router 페이지/레이아웃, 인증/권한 가드, Zustand auth store, axios/SWR API 계층, 관리자 화면, 설정 파일.
- 전체 평: 프로덕션 빌드는 통과하지만, 인증/권한/API 응답 처리 쪽에 먼저 손봐야 할 리스크가 있다.
- 최종 권고: **REQUEST CHANGES**. 특히 `Authorization` 헤더 정리, `ApiResponse.code` 검증, 권한 모델 정합성을 먼저 처리하는 것이 좋다.
- 아키텍처 상태: **WATCH**. 당장 구조가 깨진 것은 아니지만, 인증/권한/데이터 패칭 책임이 여러 곳에 퍼져 있어 기능이 늘수록 drift가 생기기 쉽다.

## 검증 결과

| 명령 | 결과 | 비고 |
| --- | --- | --- |
| `npx tsc --noEmit --incremental false` | PASS | TypeScript 타입 체크 통과 |
| `npm run build` | PASS | 네트워크 허용 상태에서 Next.js 16.2.2 Turbopack 빌드 통과 |
| `npm run lint` | FAIL | `react-hooks/set-state-in-effect` 5건, unused variable 1건 |

`npm run build`는 기본 샌드박스에서는 `next/font`가 Google Fonts를 가져오지 못해 실패했지만, 네트워크 접근을 허용하자 정상 통과했다. 즉 현재 빌드 실패 원인은 코드 컴파일 오류가 아니라 네트워크 제한이었다.

## 좋은 점

- App Router route group을 공개 영역, 보호 영역, 관리자 영역으로 나누려는 의도가 명확하다.
- access token을 `localStorage`에 persist하지 않고 메모리에만 두는 방향은 보안상 좋은 선택이다.
- refresh token은 HttpOnly cookie를 전제로 하고, axios는 `withCredentials: true`를 일관되게 사용한다.
- 401 재발급 중복 요청을 `failedQueue`로 묶어 처리하는 구조가 있어 동시 요청 폭주를 어느 정도 막는다.
- 백엔드 페이지 응답 `{ items, totalCnt, pageRows, pageNum }`을 `toPageResult()`로 변환하는 계층이 있다.
- 날짜 요청 직렬화도 `toApiLocalDate`, `toApiOffsetDateTime`으로 공통화하려는 규칙이 잡혀 있다.
- Sentry는 DSN이 없으면 초기화하지 않는 no-op 구조라 로컬/미설정 환경에서 안전하다.

## 주요 이슈

### HIGH 1. 로그아웃 후 오래된 Authorization 헤더가 남을 수 있음

- 위치: `lib/api/client.ts:91`, `store/useAuthStore.ts:32`
- 근거: refresh 성공 시 `api.defaults.headers.common['Authorization']`에 새 토큰을 넣는다. 그런데 `logout()`은 Zustand 상태만 지우고 axios default header는 지우지 않는다.
- 영향: 로그아웃 뒤에도 login/register/password-reset 같은 요청에 이전 Bearer token이 붙을 수 있다. 사용자가 바뀌거나 토큰이 폐기된 뒤에도 의도치 않은 인증 헤더가 전송될 수 있어 디버깅과 보안 양쪽에서 좋지 않다.
- 권장 수정:
  - 가장 단순한 방향은 axios default Authorization을 아예 쓰지 않고 request interceptor에서 현재 store token만 매번 주입하는 것이다.
  - token이 없으면 `config.headers.Authorization`을 명시적으로 제거한다.
  - `logout()` 또는 별도 `clearAuthSession()`에서 axios default header도 함께 삭제한다.

### HIGH 2. API 공통 응답의 `code`를 확인하지 않고 `data`만 성공 처리함

- 위치: `lib/api/types.ts:1`, `lib/api/auth.ts:41`, `lib/api/member.ts:97`, `lib/api/swr.ts:4`
- 근거: 문서상 성공 조건은 `ApiResponse.code === "00000"`인데, 대부분의 `unwrap` 함수와 SWR fetcher는 `res.data.data`만 반환한다.
- 영향: 백엔드가 HTTP 200 + 실패 code를 보내면 프론트는 성공으로 처리할 수 있다. 로그인, 예약 생성, 계약 처리, 관리자 mutation 같은 흐름에서 비즈니스 실패가 조용히 넘어갈 위험이 있다.
- 권장 수정:
  - `unwrapApiResponse(response)` 같은 공통 helper를 만든다.
  - `code !== "00000"`이면 `messageForCode(code, message)`를 포함한 typed error를 throw한다.
  - `lib/api/swr.ts`와 모든 service `unwrap`을 이 helper로 통일한다.

### HIGH 3. 권한을 백엔드 권한값이 아니라 `memberType`에서 추론함

- 위치: `lib/api/member.ts:29`, `lib/api/member.ts:61`, `lib/api/member.ts:67`, `components/manager/ManagerShell.tsx:14`
- 근거: `MemberResponseRaw`에는 `permissions`가 없고, `derivePermissions(memberType)`가 `ADMIN_USER`, `ADMIN_MASTER`, `ADMIN_DEVELOPER` 모두에게 `READ/WRITE/MANAGE`를 부여한다. 관리자 영역은 `ProtectedShell requirePermission="MANAGE"`로 보호된다.
- 영향: 프론트 UI 권한과 실제 백엔드 권한 정책이 어긋나기 쉽다. 백엔드가 최종 권한을 강제하더라도, 프론트는 메뉴와 화면 접근을 잘못 열 수 있다.
- 권장 수정:
  - `/api/v1/member/me` 또는 별도 endpoint에서 실제 `permissions[]`를 받도록 계약을 정리한다.
  - 당장 API가 없다면 `memberType` 추론은 임시 fallback으로 제한하고, 넓은 `MANAGE` 권한 부여를 줄인다.
  - route guard와 navigation visibility가 같은 ACL 정의를 공유하게 만든다.

### MEDIUM 1. `/manager/role` 페이지에 라우트 단위 master guard가 없음

- 위치: `app/(manager)/manager/role/page.tsx:15`, `constants/navigation.ts:94`
- 근거: 사이드바에서는 `회원 역할 부여` 메뉴가 `ADMIN_MASTER`에게만 보이도록 되어 있다. 그러나 페이지 자체는 별도 `ProtectedShell requireMemberType`으로 감싸지 않는다. 직접 URL 접근 시 상위 `ManagerShell`의 `MANAGE`만 통과하면 된다.
- 영향: 메뉴 숨김과 실제 페이지 접근 정책이 다르다.
- 권장 수정: `role-permission/page.tsx`처럼 페이지 자체도 `ProtectedShell requireMemberType={['ADMIN_MASTER']}`로 감싼다.

### MEDIUM 2. ESLint가 실패함

- 위치:
  - `app/(manager)/manager/reservation/page.tsx:93`
  - `app/(manager)/manager/reservation/page.tsx:300`
  - `app/(protected)/reservation/new/page.tsx:44`
  - `components/manager/RolePermissionTemplatePanel.tsx:53`
  - `components/manager/RolePermissionTemplatePanel.tsx:54`

- 근거: React 19 lint rule `react-hooks/set-state-in-effect` 오류 5건과 unused variable 경고 1건이 있다.
- 영향: CI에서 lint를 걸면 현재 상태로는 통과하지 못한다. React hooks 규칙 위반은 렌더링 cascade와 예측 어려운 fetch 흐름으로 이어질 수 있다.
- 권장 수정:
  - 읽기 데이터는 SWR 또는 공통 query hook으로 옮기는 방향이 가장 자연스럽다.

### MEDIUM 3. 인증/라우트/화면 chrome 책임이 전역에 분산됨

- 위치: `app/layout.tsx:30`, `components/auth/AuthGate.tsx:34`, `components/layout/SiteChrome.tsx:7`, `hooks/useRequireAuth.ts:21`, `components/manager/ManagerShell.tsx:12`
- 근거: root layout이 모든 페이지를 `AuthGate`와 `SiteChrome`으로 감싼다. `SiteChrome`는 `/manager` prefix를 알고 full-bleed 여부를 판단한다. 보호 정책은 `ProtectedShell`, `useRequireAuth`, `ManagerShell`, `Header`, `Sidebar`에 나뉘어 있다.
- 영향: 지금은 동작해도 정책이 늘어날수록 어디에서 접근 제어를 고쳐야 하는지 불명확해진다. public/protected/admin의 책임 경계가 route group보다 컴포넌트 조건문에 더 많이 들어가 있다.
- 권장 수정:
  - public/protected/manager route group layout이 각자 shell과 정책을 소유하게 정리한다.
  - `SiteChrome`가 `/manager` 경로 문자열을 알 필요가 없게 만든다.
  - `isAuthenticated`는 store에 별도 저장하기보다 `!!accessToken` 또는 session selector에서 파생하는 편이 drift가 적다.

### MEDIUM 4. 읽기 데이터 패칭 패턴이 페이지마다 흩어져 있음

- 위치: `lib/api/swr.ts:4`, `app/(protected)/reservation/page.tsx:18`, `app/(protected)/contract/[id]/page.tsx:18`, `app/(manager)/manager/role/page.tsx:46`
- 근거: SWR fetcher가 준비되어 있지만 주요 화면은 `useEffect + useState + 직접 service call` 패턴을 반복한다.
- 영향: loading/error/retry/cancel/cache 정책이 화면마다 달라진다. React lint 오류도 이 패턴에서 일부 발생한다.
- 권장 수정:
  - 목록/상세 조회는 SWR 기반 hook으로 통일한다.
  - mutation은 service call로 두되, 성공 후 `mutate()`로 캐시를 갱신한다.
  - 페이지 컴포넌트는 화면 상태와 사용자 입력에 집중하게 만든다.

### MEDIUM 5. 핵심 동작에 대한 테스트가 없음

- 영향: 토큰 재발급 큐, stale Authorization 제거, API envelope 검증, 날짜 직렬화, route guard 같은 중요한 동작이 회귀되어도 자동으로 잡기 어렵다.
- 권장 수정:
  - Vitest 또는 Jest 중 하나를 정해 최소 단위 테스트부터 추가한다.
  - 우선순위: `unwrapApiResponse`, `date-serializer`, `client.ts` refresh queue/logout, `derivePermissions` 또는 향후 ACL helper.

## Next.js 관점 설명

- 이 프로젝트는 App Router를 쓰지만, 대부분의 인증과 데이터 로딩은 클라이언트 컴포넌트에서 처리한다. 이 방식은 구현이 빠르고 SPA처럼 동작하지만, 서버 컴포넌트/서버 가드의 장점을 많이 쓰지는 않는다.
- 현재 보호는 주로 `ProtectedShell`이 브라우저에서 판단한 뒤 redirect하는 방식이다. 따라서 백엔드 권한 검사는 반드시 별도로 강제되어야 한다.
- `next/font/google`은 빌드 시 Google Fonts를 fetch한다. 네트워크가 막힌 CI나 사내망에서는 빌드가 실패할 수 있으므로, 필요하면 폰트 self-host 또는 CI 네트워크 허용 정책을 정해야 한다.

## 최종 판단

현재 코드는 타입 체크와 프로덕션 빌드는 통과한다. 하지만 lint가 실패하고, 인증/API/권한 계층에 운영 리스크가 있다. 프론트엔드 경험이 많지 않은 팀이라면 기능 추가보다 먼저 공통 API unwrap, 권한 정책, 인증 header 정리를 해두는 편이 이후 유지보수 비용을 크게 줄인다.

- stale `Authorization` 헤더 제거

이 단계까지 마치면 대략 **78~82점** 수준까지 올라갈 수 있다.

### 2단계: lint 100% 통과

- 현재 `npm run lint`가 실패하므로 CI 기준으로는 바로 감점이다.
- `useEffect + setState` 패턴을 SWR/query hook 또는 별도 hook으로 정리한다.
- `components/layout/Header.tsx`의 unused variable 같은 정적 경고도 제거한다.

### 3단계: 테스트 최소 세트 추가

처음부터 E2E까지 크게 갈 필요는 없다. 우선 아래 단위 테스트만 추가해도 회귀 방지 효과가 크다.

- API envelope unwrap
- token refresh queue
- logout 시 auth header 제거
- date serializer
- 권한/ACL helper

이 단계까지 마치면 **85~88점** 수준을 기대할 수 있다.

### 4단계: 권한 정책 단일화

현재는 `ManagerShell`, `Sidebar`, `ProtectedShell`, 페이지별 guard에 권한 판단이 퍼져 있다. `lib/auth/access-control.ts` 같은 곳에 route/menu 권한 정책을 모으고, 화면과 라우트가 같은 정의를 쓰게 만드는 것이 좋다.

목표는 “메뉴는 숨겼지만 URL 직접 접근은 되는” 유형의 불일치를 구조적으로 막는 것이다.

### 5단계: 데이터 패칭 표준화

조회 화면은 SWR 기반 hook으로 통일한다.

예시:

- `useMyReservations`
- `useContracts`
- `useAdminReservations`

페이지마다 loading/error/retry/cancel 로직을 반복하지 않게 만들고, mutation 성공 후에는 `mutate()`로 캐시 갱신을 표준화한다.

### 6단계: 운영 준비

- `next/font/google` 빌드 네트워크 의존성 해결: 폰트 self-host 고려
- `.env.local.example` 실제 파일 추가 또는 README 내용 정정
- Sentry 환경변수와 release 설정 확인
- CI에서 `npm run lint`, `npx tsc --noEmit --incremental false`, `npm run build` 자동 실행
